Context

교육운영팀의 하루

“교육 운영이 뭐 하는 팀인지” 잘 모르는 분도 이해하실 수 있도록,
하루를 크게 네 덩어리로 나눠 보았습니다.

1) 교육 전 – 준비

  • 연간 교육 일정·과정 기획 공유
  • 수강 대상자 선정·안내 메일 발송
  • 외부 강사 초청 및 방문·장비 안내
  • 교육장 예약, 출입·주차 사전 조율

2) 교육 중 – 운영

  • 현장 출석 확인(종이·엑셀·Forms 병행)
  • 강사/교육생 문의 실시간 응대
  • 당일 프로그램·장비 이슈 대응

3) 교육 후 – 정리

  • 출석부 정리 및 이력 반영
  • 교육 결과 보고, 개선 의견 취합
  • 강사 Fee 관련 내역 정리

4) 그 사이사이

  • “장소가 어디죠?”, “주차가 되나요?” 등 반복 문의 응대
  • 비슷한 안내 메일·문서 서식을 매번 새로 작성
  • 같은 정보를 여러 시스템에 여러 번 입력
올해는 이 반복되는 흐름 중, “안내 → 방문 → 출석” 세 구간을
AI와 가벼운 코드로 정돈해 보는 데에 초점을 맞췄습니다.

Problem

어디에서 시간이 새고 있었는지

새로운 시스템을 만들기보다는, “지금 하는 일을 어떻게 덜 힘들게 할까?”에 집중했습니다.

안내·커뮤니케이션 쪽

  • 과정마다 Word/PPT 양식을 새로 열어 안내 메일 작성
  • 문장·톤·여백이 제각각이라 검수 시간이 길어짐
  • 메일과 인쇄물 내용이 어긋나 수정 시 이중 작업 발생
  • “몇 시부터죠? 어디로 가면 되나요?” 문의가 반복

출석·결과 정리 쪽

  • 종이 출석부 서명 → 필기체·누락으로 재확인 필요
  • 출석부 → 엑셀 입력까지 행사당 10~15분 추가 소요
  • 한 번 적은 정보(과정명·일시·장소)를 여러 곳에 재입력
그래서 이번 Demo의 기준은 단순했습니다.
① 같은 정보를 여러 번 쓰지 말자
② 안내·방문·출석이 서로 떨어져 있지 않게 연결하자
③ 새 솔루션보다는, MS365·HTML·Copilot 수준에서 해결해 보자

Demo 1 – 안내 메일, 구조부터 다시 잡기

“과정마다 매번 새로 쓰는 안내 메일”을, HTML 템플릿 + 토큰 기반으로 정리했습니다.

왜 손을 댔는지

  • 과정마다 서식이 달라 담당자·수강생 입장에서 일관성이 부족
  • 메일·안내문·인쇄물이 서로 다른 내용을 담고 있어, 수정 시 이중·삼중 작업
  • 코드 없이 복붙만 하다 보니, 작은 오타·날짜 오류를 잡는 데 시간이 많이 소요

어떤 기준으로 정리했는지

  • 고정 문장과 자주 바뀌는 값(과정명·일시·장소 등)을 분리
  • 자주 바뀌는 값은 {토큰}으로 표기하여 한 번만 입력
  • Outlook / EP 메일에서 그대로 붙여넣을 수 있는 HTML 구조 유지

토큰 예시

토큰예시비고

{장소}C동 3층 Vision Hall사내 약칭 사용
{PIN}123456현장 출석 PIN

Demo 2 – 방문 안내, 한 페이지로

외부 강사님과 교육생 모두에게 “이 링크 하나만 보면 된다”는 기준점을 주고 싶었습니다.

왜 필요했는지

  • 방문 전날까지 “어디로 가야 하죠?”, “주차는 어떻게 되나요?” 문의가 반복
  • 메일 본문에서 찾기 어려운 출입 절차·운영자 연락처
  • 강사·교육생에게 안내하는 내용이 서로 조금씩 달라 혼선 발생

어떤 점을 목표로 했는지

  • LG Science Park 방문에 필요한 정보를 한 페이지에 모으기
  • 강사/교육생 모두 같은 링크를 쓰되, 문구만 다르게 안내
  • 실습 자료 업로드·장비 정보·차량 정보까지 한 번에 받기
이 데모는 “실제 강사 안내 사이트”를 축약해
발표용으로 옮겨놓은 버전이며, 실제 운영 시에는 사내 URL로 접속합니다.

Demo 2, 실제로는 이렇게 연결됩니다

  • Demo 1 안내 메일 하단에 “방문 안내 바로가기” 버튼 추가
  • 버튼 클릭 시, 지금 보실 LG Science Park 방문 안내 페이지로 이동
  • 여기서 강사는 장비·차량·교재·연락처·추가 문의를 한 번에 입력/업로드
지금부터는 이 페이지의 축약 화면을 실제로 보여드리고,
“어떤 정보가 자동으로 정리되는지”까지 같이 보겠습니다.

Demo 3 – PIN 출석, 종이 대신 공용 PC

세 번째 Demo는 출석입니다. 단순히 화면을 바꾼 것이 아니라,
“누가 언제 출석했는지”를 MS365 환경과 자연스럽게 연결하는 것이 목적입니다.

기존 방식의 불편

  • 종이 출석부에 서명 → 판독이 어려운 필기체, 누락된 서명
  • 행사당 10~15분 정도를 엑셀 입력·중복 체크에 사용
  • 사번 오기입·중복 서명으로 재확인 필요

바꾸고 싶었던 것

  • 교육생이 직접 이름·사번·PIN만 입력하면 출석이 기록
  • 운영자는 화면만 띄워두고, 세션 후에는 데이터만 확인
  • Power Automate로 Forms/Excel/SharePoint와 연동되도록 설계
이 Demo 역시, “새 시스템”이 아니라 MS365+간단한 웹 폼으로 구현 가능한 수준만을
일부러 목표로 삼았습니다.

Demo 3, 전체 흐름에서의 위치

  • Demo 1 안내 메일에서 이미 {PIN} 정보를 전달
  • Demo 2 방문 안내 페이지에서도 현장 출석 방식 안내
  • 교육 당일, Demo 3 화면을 공용 PC에 띄워
    교육생이 사전에 안내받은 PIN으로 출석

AI · Copilot

Copilot, 이렇게 썼을 때 가장 도움이 됐습니다

이 슬라이드의 목적은 “MS Copilot에 어떤 명령어를 주면 좋은 결과가 나오는지”를
교육운영 업무 기준으로 구체적으로 공유드리는 것입니다.

어떤 일을 Copilot에 맡겼는지

  • 개강 안내 메일 초안 작성 (톤·길이·대상 지정)
  • 기존 안내문·HTML 코드에서 토큰으로 묶을 부분 제안받기
  • 방문 안내 페이지 문구를 강사용/교육생용으로 다듬기

실제 사용했던 프롬프트 패턴

  • [메일] “{과정명} 개강 안내 메일 초안을, 내부 임직원 대상, 존댓말, 8줄 이내로 써줘. 일정/장소/문의처는 {토큰}으로 남겨줘.”
  • [토큰] “이 메일 내용에서 과정별로 바뀌는 값(과정명/일시/장소/대상/PIN)을 {토큰} 형식으로 표시해줘.”
  • [리뷰] “아래 HTML 메일 코드에서 인라인 스타일을 조금 정리하고, 모바일에서 깨질 만한 부분이 있으면 알려줘.”
Copilot이 만들어 준 결과물을 그대로 쓰기보다는,
“초안 + 검수 도우미”로 사용하는 것이 가장 안전하고 효율적이었습니다.
Copilot 대화 데모 (자동 반복)

오른쪽 대화창은 실제 업무에서 썼던 프롬프트 흐름을 기반으로,
Copilot과의 대화를 “움짤”처럼 재현한 것입니다.

FAQ

도입 전에 많이 하실 것 같은 질문

발표 중에는 질문 하나씩 짚어가며 설명 드리고,
자료로 보실 때는 아래처럼 한 번에 펼쳐진 상태로 읽으실 수 있도록 구성했습니다.

Q1. 코딩을 잘 몰라도 운영자가 쓸 수 있나요?
기본 HTML·코드 영역은 미리 만들어 두고,
운영자는 {과정명}, {일시}, {장소}, {PIN} 같은 토큰 값만 바꾸는 구조입니다.
Copilot 프롬프트 예시도 템플릿으로 제공해, 그대로 붙여넣고 시작할 수 있게 했습니다.
Q2. 다른 본부/조직에서도 바로 쓸 수 있나요?
가능합니다. 안내 문구·색상·로고만 바꾸면 구조는 그대로 재사용이 가능합니다.
특히 Demo 1(메일)과 Demo 3(PIN 출석)은 어떤 교육에도 쓸 수 있는 형태로 설계했습니다.
Q3. 유지보수 부담이 커지지는 않을까요?
별도 외부 솔루션이 아니라, MS365 + HTML + JavaScript 수준이라
기본 구조를 한 번 정리해 두면, 수정 시에도 한 곳만 고쳐서 전체에 반영할 수 있습니다.
Q4. 개인정보·보안 측면은 어떻게 고려했나요?
출석·방문 정보는 기본적으로 사내 MS365/Power Automate 환경에서만 처리하고,
외부 AI에는 교육 내용·서식 수준의 정보만 전달하도록 설계했습니다.
Copilot에 업로드하는 자료 역시, 사내 규정 범위 안에서만 사용하도록 가이드할 예정입니다.
Q5. 교육생 입장에서는 복잡하게 느껴지지 않을까요?
교육생이 직접 하는 행동은 크게 두 가지뿐입니다.
① 메일 또는 링크에서 방문 안내 페이지 한 번 확인
② 현장에서 공용 PC에 이름·사번·PIN 한 번 입력
오히려 종이 서명보다 단순하다는 피드백을 받았습니다.
Q6. 단계적으로 도입한다면, 어디부터 시작하는 게 좋을까요?
추천 순서는 Demo 1 → Demo 3 → Demo 2입니다.
우선 안내 메일 템플릿부터 정리한 뒤, PIN 출석으로 출석 공수를 줄이고,
마지막으로 방문 안내 페이지까지 확장하는 흐름이 부담이 적었습니다.
이 외에도 각 본부 상황에 맞춰 “우리 조직에 맞게 바꾸려면?”이 궁금하시면,
교육운영파트에 편하게 말씀 주시면 좋겠습니다.
